### 3.3 补给机制

- 单位在**无建筑图格**上经过固定时长后，无论是否行动，消耗 **1 点补给**
- 补给归零 → **单位被消灭**
- 单位进入**己方特定建筑图格** → **补给回满**

补给消耗间隔是一个**独立参数**（见 §7.10），不与行动间隔强制绑定。这意味着可能出现慢速单位在一次行动之间被补给扣死的情况——是否需要这种情况由数值设计决定。

## 四、行动调度机制

> ⚙️ 核心待定

### 4.1 问题

- 传统回合制每人每回合行动一次，无法体现速度差异
- 需求：高速单位在低速单位的行动间隔中能多次行动
- 需求：不需要固定"回合"边界

### 4.2 方案：优先队列 + 行动间隔（时间线模型）

**核心思路**：放弃回合概念，用时间线驱动行动。每个单位有"下次行动时刻"，永远让时刻最小的单位行动。

**数据结构**：最小堆

**行动间隔公式**（待定参数）：
```
行动间隔 = 基准值 / 速度
```

**示例**（基准值 = 100）：

| 速度 | 行动间隔 | 100 ticks 内行动次数 |
|------|---------|---------------------|
| 20   | 5       | 20 次               |
| 10   | 10      | 10 次               |
| 5    | 20      | 5 次                |

## 五、小队与伤害机制

### 5.1 小队规模

> ⚠️ 以下所有 HP/ATK/DEF 数值均为内部表示（×100）。显示时 ÷100。

每个单位由**小队规模**个成员组成，单位 HP 为全体成员 HP 之和：

```
单个成员 HP 上限 = 单位 HP // 小队规模
```

当单位受损、成员阵亡时，ATK 和 DEF 按存活成员比例**线性降低**：

```
当前 ATK = 配置 ATK × 存活成员数 // 小队规模
当前 DEF = 配置 DEF × 存活成员数 // 小队规模
```

### 5.2 攻击标签

| 攻击标签 | 含义 | 伤害分配 |
|----------|------|----------|
| true（AOE） | 范围攻击 | 伤害由目标**全体**小队存活成员平均分配 |
| false（集中） | 集中攻击 | 受击成员数 = 攻击方**当前**小队成员数，伤害由受击成员平均分配 |

**集中攻击（false）详解**：
- 受击成员数 = 攻击方当前存活成员数（而非小队规模）
- 伤害平均分配给每一个受击成员
- 结算后若某受击成员所受伤害 > 其 HP 上限，该成员被消灭
- **溢出伤害不产生额外影响**——消灭受击成员后，多余伤害不传递给其他小队成员

**集中攻击示例**：

| 参数 | 攻击方 A | 防御方 B |
|------|---------|---------|
| 小队规模 | 4 | 6 |
| 当前成员数 | 2 | 6 |
| 单位 HP（内部） | — | 6000 |
| 成员 HP 上限（内部） | — | 1000（显示 10.00） |

攻击方 A 对 B 造成结算伤害 20000（内部精度，显示 200.00）：
- 受击成员数 = A 当前成员数 = **2**
- 每受击成员承受 = 20000 // 2 = **10000** 伤害（内部）
- 2 名受击成员 HP 上限各 1000 → 各受 10000 伤害 → **被消灭**
- 溢出 (10000−1000)×2 = 18000 伤害，**无额外影响**
- 结果：B 剩余 4 名成员存活

## 六、伤害修正算法（DICE）

> 来源：`DICE.py`（`Mathematical_Models_Algorithms` 仓库 `/多个实体的交互迭代算法原型/`）。项目中调用时做如下适配。

### 6.1 输入构造

攻击方与防御方各取 7 个 ID，按固定顺序拼接。CSV 中的前导零予以保留。

拼接顺序：`[图格ID, 地形ID, 群落ID, 温度ID, 湿度ID, 天气ID, 单位ID]`

| 参数 | 构造方式 |
|------|---------|
| W1（entity_A） | 攻击方各 ID 字符串按顺序拼接后，**逐位拆分为个位数列表**。例：ID 为 `"01","05","02","04","02","01","01"` → 拼接 `"01050204020101"` → `[0,1,0,5,0,2,0,4,0,2,0,1,0,1]` |
| W2（entity_B） | 防御方各 ID 同理 |

### 6.2 迭代次数

不同于示例代码的固定 5 轮，项目中迭代次数由 W1 与 W2 全部元素之和决定：

```
iterations = sum(W1) + sum(W2)
若 iterations == 0，则 iterations = 1
```

14 位数字全部参与，迭代上限 ≤ 14×9×2 = 252，实际均值远低于此。修正幅度始终被 ±9.99% 封顶。

### 6.3 算法流程

1. 构造 W1、W2 并计算迭代次数
2. 运行 DICE 迭代算法，**仅取最后一轮迭代结果**（示例代码输出所有轮次，项目仅需最终结果）
3. 对结果字符串逐位相加：`"123456"` → `1+2+3+4+5+6 = 21`
4. 若和的位数 > 3，重复步骤 3，直至位数 ≤ 3
5. `a = 最终值 / 100`，例：最终值 525 → a = 5.25，修正幅度 5.25%
6. **正负判定**：取最终值的个位数，奇数 → 向下修正，偶数（含 0）→ 向上修正
7. **内部精度结算（全程整数）**：
   - `factor = 10000 + 最终值`（偶数个位）或 `10000 - 最终值`（奇数个位）
   - `最终伤害 = 基础伤害 × factor // 10000`
   - 基础伤害为 ATK 结算后的内部精度值（×100）

### 6.4 完整示例

攻击方 ID：图格=01, 地形=05, 群落=02, 温度=04, 湿度=02, 天气=01, 单位=01
防御方 ID：图格=02, 地形=03, 群落=04, 温度=01, 湿度=01, 天气=04, 单位=05

**输入构造**：
```
W1 = 拼接 "01050204020101" → [0,1,0,5,0,2,0,4,0,2,0,1,0,1]
W2 = 拼接 "02030401010405" → [0,2,0,3,0,4,0,1,0,1,0,4,0,5]
```

**迭代次数**：
```
sum(W1) + sum(W2) = 16 + 20 = 36
```

**DICE 运行**：以 W1/W2 为初始输入、迭代 36 轮，仅取最终轮输出（假设输出 `"123456"`）。

**结算**（基础伤害 = 15000）：

| 步骤 | 计算 |
|------|------|
| DICE 最终输出 | `"123456"` |
| 逐位相加 | 1+2+3+4+5+6 = **21** |
| 位数 | 2 ≤ 3，停止 |
| a | 21 / 100 = 0.21，修正幅度 0.21% |
| 个位 | 1（奇数）→ 向下修正 |
| factor | 10000 − 21 = **9979** |
| 最终伤害 | 15000 × 9979 // 10000 = **14968** |
| 显示伤害 | 149.68 → 向下取整显示 **149**（或按需四舍五入） |

## 八、地图生成机制

### 8.1 整体管线

```
球面几何 → 水域(含河流) + 水岸替换 → 地形(高程/聚落/道路/废墟) → 温度场 + 湿度场 → 群落(概率表)
```

> 水岸替换在水域生成后立即执行。废墟、聚落、道路为地形维度的后续覆盖步骤，生成规则暂时搁置。

### 8.2 球面几何

球面仅用于数据生成阶段，不进入渲染层。
- 选择 Goldberg 参数 T，得 N = 20×T 个面
- 导出：每个面的球面中心坐标（3D 单位球面）、邻接表（每个面相邻 5 或 6 个面）、经纬度（驱动温度场）
- 玩家看到的是沿邻接图 BFS 展开的平面六边形窗口

### 8.3 水域生成 + 水岸替换

首先生成水域，后续所有陆地属性均以此为基础。

1. 随机种子选定若干"洋面中心"（深海起点）
2. 从洋面中心以一定概率向邻接面扩张，形成连续的海洋区域
3. 浅海包裹深海边缘，形成海岸过渡带
4. 湖泊：陆地内部集中低洼区生成
5. 河流：从内陆随机起点沿高程梯度向海洋/湖泊方向生长

水域生成后，每个图格标记为：**深海 / 浅海 / 湖泊 / 河流 / 陆地**。

**水岸替换**：水域生成后立即执行。对于 [湖泊/浅海/深海] 与 [平原/丘陵/平缓山地/陡峭山地/普通道路/铺装道路/村庄/小镇/城市/都市/小型废墟/中型废墟/大型废墟] 的交界处，将水域侧的图格替换为水岸(11)。水岸为基础地形图格。

### 8.4 地形生成

地形为物理维度，决定移动消耗、视野、防御加成。按以下层次生成：

**第一层：水文地形**（§8.3 已确定）
- 水域：湖泊(08) / 浅海(09) / 深海(10)
- 河流：小河流(05) / 中河流(06) / 大河流(07)
- 水岸：水岸(11)

**第二层：高程 → 高地/平地**
- 对非水域图格，按噪声+山脊算法生成高程值
- 高地：丘陵(02) / 平缓山地(03) / 陡峭山地(04)，按高程阈值分配
- 平地：其余非水域图格默认为平原(01)
- 高地倾向于沿特定方向（山脊）连续分布

**第三层：人造地形**（生成规则搁置，后续覆盖）
- 道路：普通道路(12) / 铺装道路(13)
- 聚落：村庄(14) / 小镇(15) / 城市(16) / 都市(17)
- 废墟：小型废墟(18) / 中型废墟(19) / 大型废墟(20)

> 地形生成完成后，所有图格的地形 ID 已确定。群落生成（§8.8）在此基础上叠加生态属性。

### 8.5 温度场

由纬度 + 高程联合决定：

```
温度 = lat_to_temp(latitude) + elevation_penalty + noise
```

| 纬度带 | 基础温度 |
|--------|---------|
| 赤道 [0°, 15°] | 极热 / 高热 |
| 亚热带 (15°, 30°] | 热 |
| 温带 (30°, 50°] | 适中 |
| 亚寒带 (50°, 65°] | 寒 / 高寒 |
| 寒带 (65°, 90°] | 极寒 |

高程惩罚：高地比同纬度平坦地温度低 1~2 级。

### 8.6 湿度场

由水域距离 + 高程共同决定：

```
湿度 = f(距最近水域距离, 高程)
```

| 因子 | 影响 |
|------|------|
| 距水域近 | 湿度 ↑ |
| 距水域远 | 湿度 ↓ |
| 高地 | 迎风面湿 / 背风面干（简化：随机一半高地为干燥） |

## 九、视图与渲染架构

### 9.1 设计原则

不使用 3D 游戏引擎。利用球面局部的平面近似性，以**邻接图窗口 + 摩尔维特小地图**实现"全球无缝视角"。渲染层仅做显示，不涉入游戏逻辑。

### 9.2 双视图模式

同一份 Goldberg 邻接图，两种视图：

| | 策略视图 | 战术视图 |
|------|---------|---------|
| 用途 | 全局态势、资源分布 | 单位移动、战斗、建筑操作 |
| 可见范围 | 全局（缩略六边形，≤80 行） | 以当前面为中心的 BFS 窗口（玩家自行缩放） |
| 六边形尺寸 | 极小（数像素） | 大（128×128） |
| 渲染内容 | 势力涂色、建筑图标、资源点 | 地形贴图、单位、建筑、移动范围 |
| 交互 | 点击跳转区域、概览 | 单位选择、移动、攻击 |

### 9.3 渲染窗口

以当前焦点面为中心，沿邻接图 BFS 展开 N 步，渲染为平面六边形网格。窗口外扩展 1~2 步进行预渲染，玩家平移视角时增量加载。

**最大缩放约束**：玩家缩放到极限时，视口中最多显示 **80 行**六边形。对应：
- BFS 深度 = 40（从中心向任一方向展开 40 步）
- 单屏最多 ≈ 3×40² + 3×40 + 1 ≈ **4921 格**

该上限远大于最大地图规模（T=147 → 2940 格），所有规格均无需全图暴力渲染。渲染窗口随缩放级别动态调整 BFS 深度，始终控制在 4921 硬上限以内。

```
            ▲ 预渲染缓冲
        ┌───┬───┬───┐
        │ ░ │ ░ │ ░ │
        ├───┼───┼───┤
        │ ░ │ 视 │ ░ │  ← 主视图
        ├───┼───┼───┤
        │ ░ │ ░ │ ░ │
        └───┴───┴───┘
```

平面化处理：窗口面数少时球面曲率可忽略，六边形直接按 uniform 间距排列。极区附近可接受微小形变。

### 9.4 摩尔维特小地图

- 将每个面的球面经纬度映射到摩尔维特投影坐标
- 小地图上每面为一个色块点阵，色彩 = 势力归属 / 地形类型
- 点击小地图 → 快速跳转主视图焦点面
- 玩家移动主视图时实时更新当前位置指示

### 9.5 球面闭环

邻接图本身就是闭合的。从任意面出发 BFS，最终会遍历整个球面并在极区绕回。渲染层只需跟随图的拓扑，无需感知 3D 球面几何。

### 9.6 渲染尺寸与缩放

flat-top 六边形，偏移行排列。BFS 深度 d，可见行数 ≈ 2d+1。

**基础公式**：
```
垂直跨度 = (2d+1) × √3 × r
水平跨度 = (2d+1) × 2 × r
```

**以 1080p (1920×1080) 为参考**，最大 81 行时 r ≈ 7.7px：

| 可见行数 | BFS 深度 d | r (px) | 单格尺寸 (px) | 适用模式 |
|---------|-----------|--------|-------------|---------|
| 81 | 40 | 7.7 | 15×13 | 策略·全局 |
| 41 | 20 | 15.6 | 31×27 | 策略·区域 |
| 21 | 10 | 31.1 | 62×54 | 过渡 |
| 11 | 5 | 62.4 | 125×108 | 战术·部署 |
| 5 | 2 | 124.6 | 249×216 | 战术·精细 |

128×128 贴图在中级缩放起完整显示，策略缩放时降采样为像素色块，对应势力涂色。

**地图规模**：

| 规模 | T | 面数 | 策略视图可否单屏容纳 |
|------|---|------|:---:|
| 微型 | 3 | 60 | ✅ |
| 小型 | 9 | 180 | ✅ |
| 中型 | 27 | 540 | ✅ |
| 大型 | 63 | 1260 | ✅ |
| 巨型 | 147 | 2940 | ✅ |

### 9.7 策略层与战术层的衔接

- 策略层：全局单位不计入时间线，以宏观节奏批量推进（如每 N tick 统一更新补给/生产）
- 战术层：局部单位全部进入时间线堆，精细调度
- 玩家进入/退出战术视图时，在两个层级间切换单位管理模式
- 补给机制 + 人口上限自然约束单位总数
