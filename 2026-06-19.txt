## 二、地图生成机制

### 2.1 整体管线

```
球面几何 → 高程模型 → 水域(水位面: 深海/浅海/湖泊) + 陆地地形 → 河流 + 水岸替换 → 温度场 + 风场 → 湿度场(含雨影) → 群落(概率表) → 聚落/废墟 → 道路
```

> 高程模型提供连续数值基底；全局水位面在此之上一次性完成海洋、湖泊、陆地分类及陆地地形细分。河流沿高程梯度下流，终点为湖泊或海洋。风场驱动湿度分布与雨影效应。聚落/废墟/道路在群落生成后进行，受地形与群落共同约束。

### 2.2 球面几何

球面仅用于数据生成阶段，不进入渲染层。

**点阵生成（Fibonacci 球面）**：对任意面数 N，在单位球面上生成准均匀分布的点：

```
for i in range(N):
    y = 1 - 2*i/N              # 纬度分量，均匀分布 [-1, 1]
    r = sqrt(1 - y²)           # 水平半径
    φ = i × π × (3-√5)        # 黄金角螺旋
    x, z = r×cos(φ), r×sin(φ) # 球面坐标
```

**邻接构建**：对每个点，取球面上距离最近的 6 个点为邻居。多数点获得恰好 6 个邻居，极少数点可能为 5 或 7 个（与球面拓扑本质一致）。

**导出**：
- 每个点的球面中心坐标（3D 单位球面）
- 邻接表（每点相邻 5~7 个面）
- 经纬度（由 y 坐标和 φ 角导出，驱动温度场）

> N 从 300 到 30000+ **任意取值**，不受任何几何参数约束。玩家看到的是沿邻接图 BFS 展开的平面六边形窗口。

### 2.3 高程模型

高程是地图生成的核心驱动——所有地形分类、水域、河流流向、温度修正、雨影效应均以此为基础。

对全部图格在 Fibonacci 球面的 3D 坐标 `(x, y, z)` 上叠加多层噪声生成连续高程值：

```
elevation = base_noise(x, y, z) × ridge_factor + mountain_noise(...)
```

**输出**：每个图格一个 `[0, 1]` 归一化高程值。低值对应深海，高值对应山地。

高程本身不直接决定地形分类——分类由 §2.4 的全局水位面系统完成。高程仅作为连续的数值基底输入水位面判定。

> 高地倾向于沿特定方向（山脊）连续分布，由 `ridge_factor` 控制走向。

### 2.4 水域生成（全局水位面）

水域由**全局水位面**统一判定——在高程模型之上叠加若干水位阈值，一次性完成海洋、湖泊、陆地的分类。

**核心规则**：

```
对每个图格：
    if 高程 < 深海水位面        → 深海(10)
    elif 高程 < 浅海水位面      → 浅海(09)
    elif 高程 < 湖泊水位面      → 湖泊(08)
    else                       → 陆地
```

**深海水位面**：全局常数，决定海洋范围。所有低于此面的图格均为深海，形成连续的大洋区域。

**浅海水位面**：深海水位面 + 大陆架宽度。浅海包裹深海边缘，形成海岸过渡带。

**湖泊水位面**：作用于深海/浅海之外的陆地图格。**湖泊水位面不是全局常数，随纬度带变化**（纬度从 §2.2 Fibonacci 球面坐标直接导出，无需等温度/湿度计算）：

| 纬度带 | 水位面偏移 | 效果 |
|--------|-----------|------|
| 赤道 + 近赤道 [0°,20°] | +高 | 蒸发强烈，湖极少 |
| 亚热带 (20°,35°] | +中 | 偏干燥，湖较少 |
| 温带 (35°,50°] | 0 | 基准，湖数量适中 |
| 近寒带 (50°,60°] | -低 | 降水+冰蚀，湖较多 |
| 亚寒带 + 寒带 (60°,90°] | -低 | 冰蚀湖效应，湖多 |

湖泊水位面可叠加局部噪声，使湖岸线不规则（模拟自然湖岸形态）。

**优势**：海洋和湖泊共用同一套高程阈值逻辑，无需洼地检测 / 集水区 BFS / 溢出口计算 / 面积过滤。湖泊数量与面积通过调整水位面偏移量精确控制。

**陆地地形**：水域判定中落入 `else` 分支的图格按高程进一步细分：

```
else  → 陆地
    if 高程 < 丘陵阈值        → 平原(01)
    elif 高程 < 平缓山地阈值   → 丘陵(02)
    elif 高程 < 陡峭山地阈值   → 平缓山地(03)
    else                      → 陡峭山地(04)
```

> 高地（丘陵及以上）倾向于沿山脊连续分布，由高程模型中的 `ridge_factor` 控制。

**河流**：
1. 在陆地（非深海/非浅海/非湖泊）的图格中随机选取河流源头，偏好高地和山麓
2. 从源头出发，每一步向邻接面中**高程最低且低于当前**的方向流动
3. 多条支流汇合后升级：小河流(05) → 中河流(06) → 大河流(07)（按汇流数量阈值）
4. 终点：汇入湖泊、海洋，或在内陆盆地形成内流河（逐渐消失）

**水岸替换**：水域生成后立即执行，覆盖以下两种情况的水陆交界：

1. **湖海交界**：[湖泊/浅海/深海] 与 [平原/丘陵/平缓山地/陡峭山地] 的交界处 → 将水域侧图格替换为水岸(11)
2. **河流入海口**：河流汇入湖泊或海洋时，汇入点的前一格（河流末端图格）替换为水岸(11)

水岸为基础地形图格。

### 2.5 地形汇总

以上步骤完成后，所有图格的地形 ID 已确定：

| 来源 | 地形 |
|------|------|
| §2.4 水位面 | 深海(10) / 浅海(09) / 湖泊(08) / 平原(01) / 丘陵(02) / 平缓山地(03) / 陡峭山地(04) |
| §2.4 河流 + 水岸 | 小河流(05) / 中河流(06) / 大河流(07) / 水岸(11) |
| §2.11-8.12 后续 | 道路(12-13) / 聚落(14-17) / 废墟(18-20) |

> 道路、聚落、废墟在群落生成后覆盖，此时仅占位。

### 2.6 温度场

由纬度 + 高程联合决定：

```
temp_id = lat_to_temp(latitude) + elevation_penalty + noise
temp_id = clamp(temp_id, 01, 07)
```

**纬度 → 基础温度**：

| 纬度带 | 范围 | 基础温度 ID |
|--------|------|-----------|
| 赤道 | [0°, 10°] | 01（极热） |
| 近赤道 | (10°, 20°] | 02（高热） |
| 亚热带 | (20°, 35°] | 03（热） |
| 温带 | (35°, 50°] | 04（适中） |
| 近寒带 | (50°, 60°] | 05（寒） |
| 亚寒带 | (60°, 75°] | 06（高寒） |
| 寒带 | (75°, 90°] | 07（极寒） |

> 7 级温度各对应一个纬度带基线，相邻带温差为 1 级 ID。`lat_to_temp()` 在带边界附近按距离概率混合邻带基础 ID，实现纬度方向的平滑过渡。此步骤独立于后续的局部扰动。

**高程惩罚**（地形类型 → 固定偏移）：

| 地形 | offset | 说明 |
|------|--------|------|
| 平原(01) | 0 | 基准，无惩罚 |
| 丘陵(02) | +1 | 温度 ID +1，变冷 1 级 |
| 平缓山地(03) | +1 | 同上 |
| 陡峭山地(04) | +2 | 温度 ID +2，变冷 2 级 |
| 小河流(05) / 中河流(06) / 大河流(07) | 0 | 谷底效应，无额外惩罚 |
| 湖泊(08) / 浅海(09) / 深海(10) / 水岸(11) | 0 | 水域不适用高程惩罚 |
| 道路/聚落/废墟(12-20) | 0 | 人造地形无高程惩罚 |

**局部扰动**：

```
noise = random_choice([-1, 0, +1], weights=[0.15, 0.70, 0.15])
```

即 70% 概率不变，15% 变冷一级，15% 变暖一级。

**生成流程总结**：

每个地图格的温度 ID 按以下三步确定：

```
1. lat_to_temp(latitude)     → 基础温度 ID（01~07），带边界概率混合
2. + elevation_penalty       → 地形查表偏移（0 / +1 / +2）
3. + noise                   → 局部扰动 [-1, 0, +1]，15%/70%/15%
                               ↓
                          clamp(01, 07)
```

> 最终温度 ID 存入图格属性，直接参与单位的温度适性修正（§8.8）。温度在生成管线中的位置：高程 → 水域 → **温度** → 风场 → 湿度 → 群落。

### 2.7 全球风场

简化行星风带模型，纬度驱动大致风向：

| 纬度带 | 盛行风向 | 符号 |
|--------|---------|:--:|
| 赤道 + 近赤道 [0°, 20°] | 东风（信风） | ← |
| 亚热带 (20°, 35°] | 弱东风 / 静风 | ←/○ |
| 温带 + 近寒带 (35°, 60°] | 西风 | → |
| 亚寒带 + 寒带 (60°, 90°] | 东风（极地东风） | ← |

> 风向符号以赤道视角为准（→ = 自西向东）。风向叠加噪声模拟季风与气旋扰动。

**风场对湿度的影响**（见 §2.8）：空气沿主导风向移动时，遇高地抬升 → 迎风面降雨 → 背风面形成**雨影区**，湿度显著降低。

### 2.8 湿度场

由水域距离、风场、高程共同决定：

```
湿度 = base(距最近水域距离)
     + wind_moisture(风向上游的水域贡献)
     + rain_shadow(高程, 风向)  # 背风面惩罚
     + elevation_drying(高程)   # 高地降水后空气变干
     + noise
```

| 因子 | 影响 |
|------|------|
| 距水域近 | 湿度 ↑ |
| 距水域远 | 湿度 ↓ |
| 迎风面（面朝水域+风向上游） | 湿度 ↑↑ |
| 背风面（高地阻挡风向上游水汽） | 湿度 ↓↓（雨影） |
| 高地本身 | 迎风面湿 / 背风面干 |

**wind_moisture 计算流程**（具体函数待定，核心逻辑如下）：

```
# 输入：待计算湿度的陆地图格 p
# 风向来自 §2.7 风场（纬度带盛行风 + 噪声扰动）
# N = 逆风追踪最大步数（待定，默认 15）

moisture = 0
current = p
blocked = false

for step = 1 to N:
    current = current 沿逆风方向的邻接格
    if current 不存在（邻接图边界）: break

    dist = step    # 距 p 的 BFS 步数

    # 高地阻挡检测
    if current.地形 ∈ {丘陵, 平缓山地, 陡峭山地}:
        blocked = true    # 山脉阻断后续水汽

    if blocked: continue  # 跳过，不再累积

    # 水域贡献
    if current.地形 ∈ {深海, 浅海, 湖泊}:
        water_size = 该水域连通区域的面积（图格数）
        contribution = water_size / (dist² × decay_factor)
        moisture += contribution

# 归一化：累积 moisture → 湿度 ID 偏移量
   低于 dryness_threshold   → offset = -1（更干燥）
   高于 moisture_threshold  → offset = +1（更潮湿）
   介于两者之间            → offset = 0（不变）
```

**关键设计点**：

| 机制 | 效果 |
|------|------|
| `water_size`（水域面积） | 大洋输送水汽远超内陆湖泊——大陆东西岸湿度差异的根本原因 |
| `1/dist²` 衰减 | 距海越远湿度下降越快，内陆自然干燥 |
| `blocked`（高地阻断） | 山脉两侧湿度截然不同——雨影效应在水汽输送侧的数学表达 |

**示例**（西风带，风向 →）：

```
大陆西岸：逆风←追踪，步数少即触海，water_size 大 → moisture 高 → 潮湿
大陆东岸：逆风←追踪，穿越整个大陆，dist 大 + 山脉阻断 → moisture 低 → 干燥
```

> 风场使湿度不再仅取决于绝对距离：同一纬度带上，大陆西岸（迎西风）与东岸（背西风）湿度差异显著。

### 2.9 气候 → 群落概率表

群落 = 温度 ID × 湿度 ID 的派生标签，权重合计 = 100。

> 本表仅作用于启用群落的陆地图格（§2.10）。高地地形同样适用——同一山脉在不同气候带可覆盖不同群落。

#### 极热 (01)

| 湿度 | 气候 | 群落概率 (%) |
|------|------|-------------|

### 4.3 补给机制

- 单位在**无建筑图格**上经过固定时长后，无论是否行动，消耗 **1 点补给**
- 补给归零 → 当前 tick 结束后**单位被消灭**（判定在每个 tick 结束时统一执行，归零单位不进入下一 tick 的行动排队）
- 单位进入**己方特定建筑图格** → **补给回满**

补给消耗间隔是一个**独立参数**（见 §8.10），不与行动间隔强制绑定。这意味着可能出现慢速单位在一次行动之间被补给扣死的情况——是否需要这种情况由数值设计决定。

## 五、行动调度机制

> ⚙️ 核心待定

### 5.1 问题

- 传统回合制每人每回合行动一次，无法体现速度差异
- 需求：高速单位在低速单位的行动间隔中能多次行动
- 需求：不需要固定"回合"边界

### 5.2 方案：优先队列 + 行动间隔（时间线模型）

**核心思路**：放弃回合概念，用时间线驱动行动。每个单位有"下次行动时刻"，永远让时刻最小的单位行动。

**数据结构**：最小堆

**行动间隔公式**（待定参数）：
```
行动间隔 = 基准值 / 速度
```

**示例**（基准值 = 100）：

| 速度 | 行动间隔 | 100 ticks 内行动次数 |
|------|---------|---------------------|
| 20   | 5       | 20 次               |
| 10   | 10      | 10 次               |
| 5    | 20      | 5 次                |

**伪代码**：
```python
timeline = [(0, unit_id, unit) for unit in units]

while 战斗未结束:
    current_time, _, unit = heappop(timeline)
    unit.act()
    next_time = current_time + BASE / unit.speed
    heappush(timeline, (next_time, unit_id, unit))
```

**终止条件**：一方所有单位被消灭（HP 归零或补给归零）→ 战斗结束。时间线自然终止，无需外部信号。

## 六、小队与伤害机制

### 6.1 小队规模

> ⚠️ 以下所有 HP/ATK/DEF 数值均为内部表示（×100）。显示时 ÷100。

每个单位由**小队规模**个成员组成，单位 HP 为全体成员 HP 之和：

```
单个成员 HP 上限 = 单位 HP // 小队规模
```

当单位受损、成员阵亡时，ATK 和 DEF 按存活成员比例**线性降低**：

```
当前 ATK = 配置 ATK × 存活成员数 // 小队规模
当前 DEF = 配置 DEF × 存活成员数 // 小队规模
```

### 6.2 攻击标签

| 攻击标签 | 含义 | 伤害分配 |
|----------|------|----------|
| true（AOE） | 范围攻击 | 伤害由目标**全体**小队存活成员平均分配 |
| false（集中） | 集中攻击 | 受击成员数 = min(攻击方当前成员数, 防御方当前成员数)，伤害由受击成员平均分配 |

**集中攻击（false）详解**：
- 受击成员数 = min(攻击方当前存活成员数, 防御方当前存活成员数)（受击人数不超过防御方可攻击的目标数）
- 受击成员按内部编号从小到大选取（确定性选择，与 DICE 可计算性一致）
- 伤害平均分配给每一个受击成员
- 结算后若某受击成员所受伤害 > 其 HP 上限，该成员被消灭
- **溢出伤害不产生额外影响**——消灭受击成员后，多余伤害不传递给其他小队成员

**集中攻击示例 1**（攻少防多）：

| 参数 | 攻击方 A | 防御方 B |
|------|---------|---------|
| 小队规模 | 4 | 6 |
| 当前成员数 | 2 | 6 |
| 单位 HP（内部） | — | 6000 |
| 成员 HP 上限（内部） | — | 1000（显示 10.00） |

攻击方 A 对 B 造成结算伤害 20000（内部精度，显示 200.00）：
- 受击成员数 = min(2, 6) = **2**
- 每受击成员承受 = 20000 // 2 = **10000** 伤害（内部）
- 2 名受击成员 HP 上限各 1000 → 各受 10000 伤害 → **被消灭**
- 溢出 (10000−1000)×2 = 18000 伤害，**无额外影响**
- 结果：B 剩余 **4** 名成员存活

**集中攻击示例 2**（攻多防少）：

| 参数 | 攻击方 A | 防御方 B |
|------|---------|---------|
| 小队规模 | 6 | 6 |
| 当前成员数 | 6 | 2 |
| 单位 HP（内部） | — | 4000 |
| 成员 HP 上限（内部） | — | 2000（显示 20.00） |

攻击方 A 对 B 造成结算伤害 6000（内部精度，显示 60.00）：
- 受击成员数 = min(6, 2) = **2**
- 每受击成员承受 = 6000 // 2 = **3000** 伤害（内部）
- 2 名受击成员 HP 上限各 2000 → 承受 3000 伤害 → **被消灭**
- 溢出 (3000−2000)×2 = 2000 伤害，**无额外影响**
- 结果：B 全灭

### 6.3 伤害计算公式

**属性结算**（以 ATK 为例，DEF/HP/MOV 同理）：

```
结算ATK = 配置ATK × 地形修正 × 群落修正 × 天气修正 × 温度修正 × 湿度修正
```

修正值来源见 §8.5-§8.9，计算规则见 §8.9b。

**伤害计算**：

```
if 结算ATK > 结算DEF:
    造成伤害 = 结算ATK - 结算DEF + (结算DEF / 2)   = 结算ATK - 结算DEF/2
else:
    造成伤害 = 结算ATK / 2
```

| 情况 | 公式 | 含义 |
|------|------|------|
| 攻击方更强 | `ATK − DEF/2` | DEF 部分抵消，每 2 点 DEF 减 1 点伤害 |
| 攻击方偏弱 | `ATK / 2` | DEF 完全抵消正面冲击，但攻击方保留半数基础伤害（最低输出保障，防止死锁） |

> 以上均为内部精度值（×100，整除截断）。最终伤害在 DICE 修正（§7）之前确定。

## 七、伤害修正算法（DICE）

> ⚠️ DICE 作为「可控随机数生成器」使用——以双方单位与环境 ID 为种子，相同条件下始终产生相同的修正值。这为战斗引入幅度有限的波动（≤ ±9.99%），同时保持决策的可计算性：玩家可以通过掌握地形/气候/单位信息预判修正方向，而非依赖不可控的概率掷骰。
>
> 来源：`DICE.py`（`Mathematical_Models_Algorithms` 仓库 `/多个实体的交互迭代算法原型/`）。项目中调用时做如下适配。

### 7.1 输入构造

攻击方与防御方各取 7 个 ID，按固定顺序拼接。CSV 中的前导零予以保留。

拼接顺序：`[图格ID, 地形ID, 群落ID, 温度ID, 湿度ID, 天气ID, 单位ID]`

| 参数 | 构造方式 |
|------|---------|
| W1（entity_A） | 攻击方各 ID 字符串按顺序拼接后，**逐位拆分为个位数列表**。例：ID 为 `"01","05","02","04","02","01","01"` → 拼接 `"01050204020101"` → `[0,1,0,5,0,2,0,4,0,2,0,1,0,1]` |
| W2（entity_B） | 防御方各 ID 同理 |

### 7.2 迭代次数

不同于示例代码的固定 5 轮，项目中迭代次数由 W1 与 W2 全部元素之和决定：

```
iterations = sum(W1) + sum(W2)
若 iterations == 0，则 iterations = 1
```

14 位数字全部参与，迭代上限 ≤ 14×9×2 = 252，实际均值远低于此。修正幅度始终被 ±9.99% 封顶。

### 7.3 算法流程

1. 构造 W1、W2 并计算迭代次数
2. 运行 DICE 迭代算法，**仅取最后一轮迭代结果**（示例代码输出所有轮次，项目仅需最终结果）
3. 对结果字符串逐位相加：`"123456"` → `1+2+3+4+5+6 = 21`
4. 若和的位数 > 3，重复步骤 3，直至位数 ≤ 3
5. `a = 最终值 / 100`，例：最终值 525 → a = 5.25，修正幅度 5.25%
6. **正负判定**：取最终值的个位数，奇数 → 向下修正，偶数（含 0）→ 向上修正
7. **内部精度结算（全程整数）**：
   - `factor = 10000 + 最终值`（偶数个位）或 `10000 - 最终值`（奇数个位）
   - `最终伤害 = 基础伤害 × factor // 10000`
   - 基础伤害为 ATK 结算后的内部精度值（×100）

### 7.4 完整示例

攻击方 ID：图格=01, 地形=05, 群落=02, 温度=04, 湿度=02, 天气=01, 单位=01
防御方 ID：图格=02, 地形=03, 群落=04, 温度=01, 湿度=01, 天气=04, 单位=05

**输入构造**：
```
W1 = 拼接 "01050204020101" → [0,1,0,5,0,2,0,4,0,2,0,1,0,1]
W2 = 拼接 "02030401010405" → [0,2,0,3,0,4,0,1,0,1,0,4,0,5]
```

**迭代次数**：
```
sum(W1) + sum(W2) = 16 + 20 = 36
```

**DICE 运行**：以 W1/W2 为初始输入、迭代 36 轮，仅取最终轮输出（假设输出 `"123456"`）。

**结算**（基础伤害 = 15000）：

| 步骤 | 计算 |
|------|------|
| DICE 最终输出 | `"123456"` |
| 逐位相加 | 1+2+3+4+5+6 = **21** |
| 位数 | 2 ≤ 3，停止 |
| a | 21 / 100 = 0.21，修正幅度 0.21% |
| 个位 | 1（奇数）→ 向下修正 |
| factor | 10000 − 21 = **9979** |
| 最终伤害 | 15000 × 9979 // 10000 = **14968** |
| 显示伤害 | 显示 HP 变动量 = 受攻击前 HP − 受攻击后 HP（均为 ÷100 显示值）；例：防御方 HP 从 500.00 → 350.32，伤害量自明，无需单独显示取整 |

## 八、配置表规范

地图、建筑、单位均通过**配置表**控制，保证可扩展性和泛用性。所有核心游戏数据不硬编码，统一从配置表读取。

> ⚠️ 建筑和单位的「阵营」「类别」「子类」均为**分类标签**，仅用于识别和筛选，不属于对象的运行时属性。

### 8.1 地图图格

每个图格包含六个属性，属性值均为对应环境参数的**数字 ID**：

| 属性 | 类型 | 值示例 |
|------|------|--------|
| 图格ID | int | 唯一标识 |
| 地形 | int | 01 / 02 / 08 ...（物理属性：移动/视野/防御） |
| 群落 | int | 01 / 04 / 05 ...（生态属性，空时表示该图格不启用群落） |
| 天气 | int | 01 / 05 ... |
| 温度 | int | 04 / 01 ... |
| 湿度 | int | 01 / 02 / 03 |
| 资源 | int | 02 / 03 ...（该属性空时表示该图格无资源） |

> 图格属性对建筑和单位的影响，由**建筑配置表**和**单位配置表**分别控制。地图层不定义影响规则，只定义"这里是什么"。

### 8.2 建筑配置表

> 建筑名称即为唯一 key，类别与子类为分类标签。

| 字段 | 类型 | 说明 |
|------|------|------|
| 建筑ID | key | 唯一标识 |
| 名称 | string | 同时是建筑的显示名称 |
| 类别 | 标签 | 经济/生产/军事（仅用于分类识别） |
| 子类 | 标签 | 资源生产/科研/人口上限/训练/…（仅用于分类识别） |
| 建造条件 | dict | 所需地形/资源/科技等前置条件 |
| 效果 | string | 描述性文本标签（具体功能及实现方式另行设计） |

### 8.3 单位配置表

> 单位名称即为唯一 key，阵营/类别/子类为分类标签。

| 字段 | 类型 | 说明 |
|------|------|------|
| 单位ID | key | 唯一标识 |
| 名称 | string | 同时是单位的显示名称 |
| 所属阵营 | 标签 | 势力归属（仅用于分类识别） |
| 类别 | 标签 | 陆地/水中/空中（仅用于分类识别） |
| 子类 | 标签 | 步兵/骑兵/机械化/舰船/…（仅用于分类识别） |
| ATK | int | 攻击力（内部 ×100，显示 ÷100） |
| DEF | int | 防御力（内部 ×100，显示 ÷100） |
| HP | int | 生命值（内部 ×100，显示 ÷100） |
| MOV | int | 移动力 |
| SPD | int | 速度 |
| VIS | int | 视野（可侦察的 BFS 步数） |
| 小队规模 | int | 单位由若干小队成员组成 |
| 攻击标签 | bool | true = AOE；false = 集中攻击 |
| 地形适性 | — | 见 §8.5 |
| 群落适性 | — | 见 §8.6 |
| 天气适性 | — | 见 §8.7 |
| 温度适性 | — | 见 §8.8 |
| 湿度适性 | — | 见 §8.9 |
| 攻击范围 | int-int (min,max) | 最小-最大攻击距离 |
| 生产消耗 | - | 训练所需各资源数量（key=资源ID），见 §8.4 |
| 生产时间 | int | 训练所需 tick 数 |
| 补给容量 | int | 最大补给值 |

### 8.4 单位生产配置表

每种资源消耗一行，同一单位可有多行：

| 字段 | 类型 | 说明 |
|------|------|------|
| 单位ID | int | 对应单位 |
| 名称 | string | 单位显示名称 |
| 消耗资源ID | int | 所需资源 ID |
| 消耗资源名称 | string | 资源显示名称 |
| 消耗数量 | int | 消耗的资源数量（正整数，单位由资源类型隐含决定） |

### 8.5 地形适性配置表

每种地形一行，列为该地形下的各属性修正值（20 种地形）：

| 单位ID | 名称 | 地形 | ATK | DEF | MOV | VIScover |
|--------|------|------|-----|-----|-----|-----|

> 空值（`—`）= 默认 1.0。MOV=0 表示该单位不可进入此地形势。VIScover 为布尔值：0 = 遮蔽 / 1 = 穿透，默认值 = 1。未列出的地形 × 单位组合默认全部属性 = 1.0。

### 8.6 群落适性配置表

每种群落一行，列为该群落下的各属性修正值（10 种群落）：

| 单位ID | 名称 | 群落 | ATK | DEF | MOV | VIScover |
|--------|------|------|-----|-----|-----|-----|

> 以上数值均为示例，非最终实装。空值 = 默认 1.0。VIScover 为布尔值：0 = 遮蔽 / 1 = 穿透，默认值 = 1。未列出的群落 × 单位组合默认全部属性 = 1.0。**最终修正 = 地形修正 × 群落修正 × 天气修正 × 温度修正 × 湿度修正。**

### 8.7 天气适性配置表

每种天气一行，列为该天气下的各属性修正值（7 种天气）：

| 单位ID | 名称 | 天气 | ATK | DEF | MOV | VIScover |
|--------|------|------|-----|-----|-----|-----|

> 以上数值均为示例，非最终实装。空值 = 默认 1.0。VIScover=0 表示遮蔽 / VIScover=1 表示穿透，默认值 = 1。未列出的天气 × 单位组合默认全部属性 = 1.0。

### 8.8 温度适性配置表

每种温度一行，列为该温度下的各属性修正值（7 种温度）：

| 单位ID | 名称 | 温度 | ATK | DEF | MOV | VIScover |
|--------|------|------|-----|-----|-----|-----|

> 以上数值均为示例，非最终实装。空值 = 默认 1.0。VIScover=0 表示遮蔽 / VIScover=1 表示穿透，默认值 = 1。未列出的温度 × 单位组合默认全部属性 = 1.0。

### 8.9 湿度适性配置表

每种湿度一行，列为该湿度下的各属性修正值（3 种湿度）：

| 单位ID | 名称 | 湿度 | ATK | DEF | MOV | VIScover |
|--------|------|------|-----|-----|-----|-----|

> 以上数值均为示例，非最终实装。空值 = 默认 1.0。VIScover=0 表示遮蔽 / VIScover=1 表示穿透，默认值 = 1。未列出的湿度 × 单位组合默认全部属性 = 1.0。

### 8.9b 修正值计算规则

所有五维适性修正以乘法叠加，按属性类型分两条路径结算：

```
raw = 配置值 × 地形修正 × 群落修正 × 天气修正 × 温度修正 × 湿度修正
```

**ATK / DEF**（浮点连续值，内部 ×100 精度）：

```
最终值 = floor(raw)        ← 截断取整，与 ×100 整数算术一致
```

**MOV**（离散整数，单位可移动的六边形格数）：

```
raw = 配置 MOV × ∏修正

if raw == 0:
    MOV = 0                         # 零值阻断：不可进入
else:
    MOV = max(1, floor(raw))        # 保底 1 格
```

**VIScover**（布尔值）：

VIScover 的最终值由五维 VIScover 修正的**逻辑或**判定：

```
若 地形.VIScover == 0 OR 群落.VIScover == 0 OR 天气.VIScover == 0 OR 温度.VIScover == 0 OR 湿度.VIScover == 0
    → 视野被遮蔽，该方向 BFS 截断
否则 → 视野穿透
```

> 未配置的组合项默认值 = 1（穿透），不参与遮蔽判断。

## 九、视图与渲染架构

### 9.1 设计原则

不使用 3D 游戏引擎。利用球面局部的平面近似性，以**邻接图窗口 + 摩尔维特小地图**实现"全球无缝视角"。渲染层仅做显示，不涉入游戏逻辑。

### 9.2 双视图模式

同一份邻接图，两种视图：

| | 策略视图 | 战术视图 |
|------|---------|---------|
| 用途 | 全局态势、资源分布 | 单位移动、战斗、建筑操作 |
| 可见范围 | 全局（缩略六边形） | 以当前面为中心的 BFS 窗口（玩家自行缩放） |
| 六边形尺寸 | 极小（数像素） | 大（128×128） |
| 渲染内容 | 势力涂色、建筑图标、资源点 | 地形贴图、单位、建筑、移动范围 |
| 交互 | 点击跳转区域、概览 | 单位选择、移动、攻击 |

### 9.3 渲染窗口

以当前焦点面为中心，沿邻接图 BFS 展开 N 步，渲染为平面六边形网格。窗口外扩展 1~2 步进行预渲染，玩家平移视角时增量加载。

**缩放约束**：缩放上下限待功能实装后根据实际体验调整，当前不做硬性限制。渲染窗口随缩放级别动态调整 BFS 深度，视口外仅预渲染 1~2 步缓冲。

```
            ▲ 预渲染缓冲
        ┌───┬───┬───┐
        │ ░ │ ░ │ ░ │
        ├───┼───┼───┤
        │ ░ │ 视 │ ░ │  ← 主视图
        ├───┼───┼───┤
        │ ░ │ ░ │ ░ │
        └───┴───┴───┘
```

平面化处理：窗口面数少时球面曲率可忽略，六边形直接按 uniform 间距排列。极区附近可接受微小形变（量化容差标准待定，见 §10.6）。

### 9.4 摩尔维特小地图

- 将每个面的球面经纬度映射到摩尔维特投影坐标
- 小地图上每面为一个色块点阵，色彩 = 势力归属 / 地形类型
- 点击小地图 → 快速跳转主视图焦点面
- 玩家移动主视图时实时更新当前位置指示

### 9.5 球面闭环

邻接图本身就是闭合的。从任意面出发 BFS，最终会遍历整个球面并在极区绕回。渲染层只需跟随图的拓扑，无需感知 3D 球面几何。

### 9.6 渲染尺寸与缩放

flat-top 六边形，偏移行排列。BFS 深度 d，可见行数 ≈ 2d+1。

**基础公式**：
```
垂直跨度 = (2d+1) × √3 × r
水平跨度 = (2d+1) × 2 × r
```

**以 1080p (1920×1080) 为参考**，最大 81 行时 r ≈ 7.7px：

| 可见行数 | BFS 深度 d | r (px) | 单格尺寸 (px) | 适用模式 |
|---------|-----------|--------|-------------|---------|
| 81 | 40 | 7.7 | 15×13 | 策略·全局 |
| 41 | 20 | 15.6 | 31×27 | 策略·区域 |
| 21 | 10 | 31.1 | 62×54 | 过渡 |
| 11 | 5 | 62.4 | 125×108 | 战术·部署 |
| 5 | 2 | 124.6 | 249×216 | 战术·精细 |

128×128 贴图在中级缩放起完整显示，策略缩放时降采样为像素色块，对应势力涂色。

**地图规模**（N 为总面数，任意取值）：

| 规模 | 面数 N | 策略视图可否单屏容纳 |
|------|--------|:---:|
| 微型 | 300 | ✅ |
| 小型 | 600 | ✅ |
| 中型 | 1200 | ✅ |
| 大型 | 3000 | ✅ |
| 巨型 | 10000 | ✅ |
| 史诗 | 30000 | ✅ |

> N 在 [300, 30000+] 范围内**连续自由取值**，不受几何参数约束。实际可玩面积通过水域率进一步灵活调节。

### 9.7 策略层与战术层的衔接

- 策略层：全局单位不计入时间线，以宏观节奏批量推进（如每 N tick 统一更新补给/生产）
- 战术层：局部单位全部进入时间线堆，精细调度
- 玩家进入/退出战术视图时，在两个层级间切换单位管理模式
- 补给机制 + 人口上限自然约束单位总数
