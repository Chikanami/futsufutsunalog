## 前期开发历程 (2026-07-09 ~ 2026-07-16)

### 一、项目起源

本项目最初设计为 C# WinForms 桌面应用，面向局域网内多终端使用。经过多次架构调整最终确定为 **Python FastAPI Web 应用 + 浏览器前端**。

**架构演变路径**：

| 阶段 | 方案 | 废弃原因 |
|------|------|----------|
| 初始 | 桌面应用 + 局域网 | 发现存在异地项目，LAN 不可达 |
| 调整 | 桌面应用 + 花生壳内网穿透 | 涉军场景不适合第三方穿透；报价超预算 |
| 最终 | **Web 应用** + 直连/隧道兼容 | 零客户端安装、更新即生效、同时兼容 LAN 和异地 |

### 二、技术决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 后端语言 | Python (FastAPI) | 主机已有 Python 环境，学习成本低 |
| 前端 | 原生 HTML/CSS/JS | 零构建、零依赖，浏览器即运行时 |
| 数据库 | SQLite (WAL 模式) | 单机部署，无需数据库服务进程 |
| 图数据库 | Neo4j (已运行) | 报价明细的关系查询场景 |
| 认证 | JWT (python-jose) | 无状态，Token 12h 有效期 |
| 密码存储 | 明文 | 内网环境，运维便利性优先 |

### 三、数据库

### 一、设计符合性审查

对照设计文档逐项审查代码，结论：主流程链路完整（批次上传 → 生成临时询价档案 → 录入校验 → 复核 → 全部复核后迁移正式档案），但存在 2 处功能性 Bug、4 处设计偏差、若干内部一致性问题。

### 二、缺陷修复（P0–P2）

**P0 — 功能性 Bug**
- `server.py`：新增 `GET /uploads/{path}` 路由，修复复核界面照片 404（原路由缺失导致核心复核环节不可用）；含路径穿越防护

**P1 — 设计符合性**

- EAN 校验由 `LIKE '%..%'` 子串匹配改为精确匹配（消除短条码误匹配长条码的风险），并将命中批次传入提交流程定位"该批次"的待录入行
- 未复核统计与认领口径统一：均排除预生成空行（`created_by IS NOT NULL`）+ 按 project 过滤

**P2 — 完善项**

- 批次上传支持管理员指定 `batch_id` / `batch_name` / `inquiry_time`（原全部自动生成）

### 三、新增功能

### 五、Bug 修复（前端）

- **录入页预览图不显示**：`style.display=''` 置空内联样式后回落到 CSS 类的 `display:none`，预览框永不显示；改为 `display='block'`。（教训：同页面 CSS 类隐藏与内联隐藏混用导致 `style.display=''` 行为不一致）

### 六、运维事项与已知问题

1. **热重载不可靠（重要）**
   - 服务以 `Start-Process -WindowStyle Hidden` 后台启动时，watchfiles 能检测到变更并打印 `Reloading...`，但旧 worker 无法被终止（Windows 下向无控制台子进程发信号失败），重载卡死、继续运行旧代码
   - **约定：每次改动 Python 代码后手动重启服务**；静态文件（HTML/JS）即改即生效无需重启
   - 重启方式：`taskkill /PID <主进程> /T /F` 后经 PowerShell `Start-Process` 拉起，日志重定向至 `data/server.log` / `data/server.err.log`
   - 曾因此出现两次"改了代码但不生效"的假象（模板路由 404、批次内容路由 404）

2. **watchfiles 已安装**，在交互式终端手动运行 `python server.py` 时热重载可正常工作

3. **测试资源**：`test_images/` 下有 4 张测试图（2 张模拟价签图含条码 6901234567890 / 6909876543210，2 张空白图）
