### 二、事故记录：服务多实例叠加（重要教训）

**现象**：重启后新路由仍随机 404。排查发现 **4 套服务进程同时监听 8088**（含一套 07-16 的旧 server.py 单体架构进程），Windows 允许多进程绑定同端口，请求被随机分配到新旧 worker。

**根因**：多轮重启使用的 PowerShell 命令 `ForEach-Object { Stop-Process -Id $_.ProcessId }` 中 `$_` 在 bash 双引号内被展开为空，Stop-Process **静默失败**，旧进程从未被杀掉，每次"重启"实为叠加新实例。

**处置**：taskkill 显式 PID 清掉全部 8 个进程（4 主 + 4 worker），干净启动单实例；连续 10 次探测确认路由稳定 401。

**规范（今后重启必须遵守）**：
1. 先 `wmic process where "name like '%python%'" get ProcessId,CommandLine` 列出全部进程
2. 用 `taskkill /PID <pid> /T /F` 显式终止（禁用含 `$_` 的 PowerShell 管道，bash 转义会破坏它）
3. 启动后**连续多次**探测新路由，确认无残留实例接管请求
4. 验证 python 进程数应为 2（reloader + worker）

### 一、users.project_ids / permissions JSON 列 → 关联表

**背景**：`users` 表的 `project_ids`（JSON 数组存项目 ID）和 `permissions`（JSON 数组存权限）使用 `json.loads/dumps` 维护，存在以下问题：
- JSON 列无法建立外键约束，数据完整性靠应用层保证
- `project_ids LIKE '%"pid"%'` 查询脆弱且无法走索引
- 权限查询需要全表 JSON 解析

**变更**：新建两张关联表：

| 新表 | 结构 | 说明 |
|------|------|------|
| `user_projects` | `user_id FK → users ON DELETE CASCADE, project_id FK → project_list ON DELETE CASCADE, PK(user_id, project_id)` | 用户-项目多对多 |
| `user_permissions` | `user_id FK → users ON DELETE CASCADE, permission TEXT, PK(user_id, permission)` | 用户权限一对多 |

**迁移** `_migrate_user_relations()`（幂等）：检测 `users` 表仍存在旧 JSON 列时搬运数据至关联表（INSERT OR IGNORE），旧列保留不删。`delete_user` 依赖 `ON DELETE CASCADE` 自动清理关联记录。

**代码重构**：`users/db.py` 全部读写改用 JOIN `user_projects` + `user_permissions`，废弃 `_ids_to_names(ids_json, ...)` 辅助函数，新增 `_get_user_projects(conn, user_id)` / `_get_user_permissions(conn, user_id)`；对外返回结构不变（`projects` 列表 + `permissions` 列表）。

**影响**：`plans/db.delete_project_config` 的用户引用检查从 `project_ids LIKE '%"pid"%'` 改为 `SELECT COUNT(*) FROM user_projects WHERE project_id = ?`。

### 十三、计划外询价条目

**背景**：录入员拍照时可能遇到不在询价计划中的商品，原流程直接 400 拒绝，但这些照片和信息有保留价值且需要复核防数据污染。

**设计**：不拒绝，标记为计划外（`is_extra=1`），同样进入正常复核流程。

**数据库**：`inquiry_temp` 和 `inquiry_records` 新增 `is_extra INTEGER DEFAULT 0` 列（`_migrate_is_extra` 幂等迁移）。

**录入流程**：
- 条码在计划内 → `is_extra=0`，正常录入
- 条码不在计划中 → `is_extra=1`，仍录入并进入复核
- 前端 toast 提示"仍将进入复核"

**复核流程不变**：计划外条目与计划内条目走相同的认领→核对→确认/修正流程，复核面板显示"（计划外）"标记。

**批次统计**：
- `list_batches` / `list_active_batches` 新增 `extra_count`
- `get_batch_stats` 返回 `extra` 计数
- 管理后台批次表格新增"计划外"列，点击数值可弹窗查看详细列表
- 批次内容弹窗支持 `?filter=extra` 筛选，计划外条目显示橙色"计划外"标签

**覆盖文件**：`core/database.py`、`inquiry/db.py`、`inquiry/routes.py`、`plans/db.py`、`plans/routes.py`、`admin/admin.html`、`admin/plan.js`、`inquiry/entry.js`、`inquiry/review.js`

### 十八、用户状态服务端持久化

**服务端状态表**：`user_session_state(user_id PK, state_type, state_data JSON, batch_id, updated_at)`，每个用户一条记录，INSERT OR REPLACE 语义。

**保存链路**：`uploadAndShow` 末尾 `await saveEntryStateAsync()` 确保落盘后再继续；`beforeunload` 事件 `fetch + keepalive` 保证页面关闭前发出；`_exitFlag` 防退出/切换计划时覆盖清除。

**恢复链路**：`setMode → restoreEntryState()` 从服务端加载状态，恢复 `uploadedFiles` 列表和 `currentIndex`。File 对象无法持久化，改为全部文件上传完毕后才允许录入。

**清理**：退出按钮 → `POST /api/state/clear` → 解锁复核条目 + 删除记录；复核完成自动清除。
